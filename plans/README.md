## Not turned into plans (flagged during audit, needs a product call)

- **`TapSparkles` fires on every tap app-wide** (`components/ui/TapSparkles.tsx`) — a `document`-level click listener spawns a decorative particle on every button/link press in the entire app, which puts it in the "100+ times/day → no animation, ever" frequency tier per this repo's own audit rules. It's clearly a deliberate charm feature (dedicated component, not an accident), so this plan set does not remove or scope it down — only gates it under `prefers-reduced-motion` (plan 006). If you want it scoped down further (e.g. suppressed during rapid trivia-answering on `/play` and `/board` but kept on nav/idle taps), that's a product decision worth deciding explicitly before writing a plan for it.
- **Framer Motion `x`/`y`/`scale` shorthand props** are used throughout (`NavRow`, `Card`, `Celebration`, `PageTransition`, `StaggerItem`, `Tooltip`, `TapSparkles`) instead of full `transform` strings, which per the audit playbook aren't independently hardware-accelerated. Not planned: every animated surface in this app renders small element counts (particle bursts ≤6, list stagger typically <10 rows, no virtualized/dense grids), so the realistic performance impact is negligible today. Revisit only if a future screen renders many concurrently-animated elements.

## Recommended execution order

1. **006** (reduced-motion foundation) and **008** (transition-all cleanup) — independent, mechanical, no dependencies on anything else. Good first PR.
2. **007** (Squiggle) — independent of everything else (touches a file none of the others do).
3. **001**, **002**, **005** — independent quick wins, no shared files with each other or with 003/004.
4. **003** then **004**, in that order — both touch `app/(protected)/categories/page.tsx`. Plan 003 restructures the row wrapper (adds the `AnimatePresence`/`motion.div` list wrapper); plan 004 edits content nested inside that same row (the delete-confirm swap). Running 003 first means 004's edits land inside the already-updated structure; running them out of order risks a merge conflict if executed by separate agents/worktrees.

## Known file overlaps (avoid parallel execution on these)

- `app/(protected)/categories/page.tsx`: plans **003** and **004** (see ordering above).
- `app/(protected)/play/page.tsx`: plans **002** (progress stars, lines ~186-201) and **008** (submit button, line ~127) touch disjoint regions of the same file — safe to run sequentially, but do not dispatch both to separate agents/worktrees in parallel, since a merge of two diffs against the same file needs a human/tool pass regardless.
- `components/ui/Squiggle.tsx` is deliberately untouched by plan 006 and left entirely to plan 007, to avoid two plans editing the same three lines.
